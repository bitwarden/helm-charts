## Critical Rules

- **NEVER** commit secrets or credentials: Installation IDs, keys, passwords must be provided via Kubernetes secrets or CSI secret providers
- **NEVER** skip schema regeneration: After modifying values.yaml, always run `helm schema` to regenerate the JSON schema
- **NEVER** use deprecated fields: Use `image.repository` instead of `image.name`
- **ALWAYS** use the lookup pattern for secret persistence: Check for existing secrets before generating new ones in Helm hooks
- **ALWAYS** follow version resolution pattern: Component tag → coreVersionOverride → chart default
- **ALWAYS** add helm-unittest tests for new components or template logic changes
- **ALWAYS** use documentSelector in tests when templates produce multiple Kubernetes resources

## Project Structure

- **Charts**:
  - `/charts/self-host/` - Full Bitwarden self-hosted deployment (API, Identity, Web, Admin, SSO, etc.)
  - `/charts/sm-operator/` - Secrets Manager Kubernetes Operator
- **Templates**: `/charts/*/templates/` - Kubernetes manifests and Helm hooks (pre-install, post-install)
- **Tests**: `/charts/*/tests/` - helm-unittest test files
- **Scripts**: `/scripts/` - Local development with Kind cluster
- **CI Values**: `/charts/*/ci/test-values.yaml` - Test configuration for CI

## Common Commands

**Linting**:
```bash
ct lint --charts charts/self-host
ct lint --charts charts/sm-operator
```

**Schema Generation** (required after values.yaml changes):
```bash
helm plugin install https://github.com/dadav/helm-schema
cd charts/<chart-name>
helm schema -k additionalProperties --skip-auto-generation required
```

**Testing**:
```bash
helm plugin install https://github.com/helm-unittest/helm-unittest
helm unittest charts/self-host
helm unittest charts/sm-operator
```

**Local Development**:
```bash
# Full Kind cluster setup
./scripts/setup.sh all

# Or individual steps
./scripts/setup.sh create-cluster
./scripts/setup.sh setup-cluster
./scripts/setup.sh install-self-host
```

**Template Rendering**:
```bash
helm template charts/self-host --values charts/self-host/ci/test-values.yaml
helm template charts/sm-operator --values charts/sm-operator/ci/test-values.yaml
```

## Development Workflow

**Before making changes**:
- Read existing templates to understand patterns (version resolution, secret persistence, CSI volumes)
- Check `helpers.tpl` for reusable template functions
- Review existing tests to understand test patterns

**When adding new components**:
1. Add values configuration to `values.yaml`
2. Create helper template in `helpers.tpl`
3. Create component template (Deployment + Service)
4. Add ingress path if needed
5. Write helm-unittest tests
6. Regenerate schema: `helm schema -k additionalProperties --skip-auto-generation required`
7. Validate: `ct lint` and `helm unittest`

**Pre-commit checklist**:
- [ ] Schema regenerated after values.yaml changes
- [ ] Tests added/updated for new functionality
- [ ] Linting passes: `ct lint --charts charts/<chart-name>`
- [ ] Unit tests pass: `helm unittest charts/<chart-name>`
- [ ] Templates render correctly with test values
- [ ] Breaking changes documented

### Key Patterns

**Version Resolution**:
```yaml
# Priority: component.tag > coreVersionOverride > chart default
image: "{{ .Values.component.api.image.repository }}:{{ .Values.component.api.image.tag | default .Values.general.coreVersionOverride | default (include "bitwarden.coreVersionDefault" .) }}"
```

**Secret Persistence** (in pre-install hooks):
```yaml
{{- $currentSecret := ( lookup "v1" "Secret" .Release.Namespace ( include "bitwarden.keyssecret" . ) ) }}
{{- if $currentSecret }}
  # Use existing values to prevent regeneration on upgrade
  globalSettings__internalIdentityKey: {{ index $currentSecret.data "globalSettings__internalIdentityKey" }}
{{- else }}
  # Generate new random value for first install
  globalSettings__internalIdentityKey: {{ (randAlphaNum 64) | b64enc | quote }}
{{- end }}
```

**CSI Secret Volumes**:
```yaml
{{- if .Values.secrets.secretProviderClass }}
volumes:
  - name: secrets-store-inline
    csi:
      driver: secrets-store.csi.k8s.io
      readOnly: true
      volumeAttributes:
        secretProviderClass: {{ .Values.secrets.secretProviderClass }}
{{- end }}
```

**Conditional Resources**:
```yaml
{{- if .Values.component.scim.enabled }}
# SCIM resources only created when enabled
{{- end }}
```

## Security Requirements

- **TLS Required**: Always configure TLS for ingress (Let's Encrypt cluster issuers supported)
- **Secret Management**: Support both Kubernetes secrets and CSI secret providers (Azure Key Vault, AWS Secrets Manager, etc.)
- **Identity Certificates**: Auto-generated by pre-install hooks and persisted across upgrades
- **Service Accounts**: Proper RBAC for pre/post-install hooks and pod access to secrets
- **No Secrets in Values**: Installation IDs, keys, passwords, connection strings must never be in values.yaml

## Anti-Patterns

**DO**:
- Use `lookup` to check for existing secrets before generating new ones
- Provide `documentSelector` in tests when templates produce multiple documents
- Test with `ci/test-values.yaml` to ensure templates render correctly
- Use CSI secret providers for production deployments
- Stage specific files when committing (avoid `git add .` to prevent secrets)

**DON'T**:
- Hardcode secrets or credentials in values.yaml
- Skip schema regeneration (CI will fail)
- Use `image.name` field (deprecated)
- Forget to add ingress paths for new components
- Use Helm hooks without proper delete policies (`helm.sh/hook-delete-policy`)
- Assume database is always deployed (check `database.enabled`)
